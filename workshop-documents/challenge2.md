# 2.アプリケーションを調査する

## このチャレンジで体験できること

* Agentモードを使用したアプリケーションの調査
* コードからドキュメントを生成する
* Mermaidなどを利用しながらグラフィカルなドキュメントを生成できる
* submoduleで提供されている情報を参照できる

## 目的

アプリケーションの構造がわかりません。モダナイゼーションを進めていくうえでアプリケーションの構造を把握することは非常に重要です。現在のアプリケーションの状態を把握しドキュメントとして残し、人とAIが活用できる資材として残します。マイクロサービスもどきの構造や内部で使用されているフレームワーク、データ構造などを明らかにしていきます。

## 手段

GitHub Copilotを使用してアプリケーションの構造を明らかにしていきます。特定された情報をドキュメント化すれば後任の方々やAIのためになります。

ヒントになりそうなキーワード
* Mermaid記法:シーケンス図の作成などを依頼したい場合に
* 利用しているポートの情報
* 初期データ:DB初期化のためのデータがあるようです
* ユニットテスト/インテグレーションテスト/e2eテスト:テストのひな型はすでに存在するようです
* サービス間連携:どのような手法をとっているかわかりませんが、このアプリケーションは他サービスと連携して一つのアプリとして提供されているようです
* /docs: リポジトリ内のアプリをここに格納しようとしたのが見て取れます

## Sample Prompt（Agent Mode）

``` markdown
このサービスについてまとめて、 `/docs` にドキュメントとしてまとめてください。
下記の内容をまとめてください。同一のファイルではなくテーマごとに個別のファイルにしてください。
ドキュメント索引がわかりやすいようにルートのREADME.mdにIndexを追記してください。
ドキュメントのフォーマットや記載内容については共通規約の `develop-standard` の内容を加味してください。
<!-- 以降は調整など好きに構成してください。生成されるドキュメントの種類が変わります -->
- 各サービスの役割、機能、提供されているWebAPIの概要
- サービス間の通信経路
- ユーザー利用時のシーケンス図（Mermeaid記法でOKです）
- テーブル構造のEF図（Mermeaid記法でOKです）
  - サービスごとに個別のDBとなっているためその表現がわかりやすく表記してください。
- 画面フロー図（Mermeaid記法でOKです）
- 初期Seedデータの内容について
    - e2eテスト、Integrationテスト時に使用するSeedデータも含め
```
